# Test Driven Development (TDD) Skill

在执行具体的编码任务时，必须严格遵守此流程。

## 目标
通过测试驱动的方式编写高质量、可维护的代码，确保功能符合预期且无回归。

## 流程指令 (Red-Green-Refactor Loop)

### 1. 🔴 Red (编写失败测试)
*   **动作**：根据当前任务的需求，编写一个新的测试用例。
*   **原则**：不要先写实现代码。
*   **验证**：运行测试，确认它**失败**（并且是由于预期的原因失败，如“函数未定义”或“断言错误”）。
*   **输出**：向用户展示测试失败的日志。

### 2. 🟢 Green (变绿)
*   **动作**：编写实现该功能所需的**最小代码量**。
*   **原则**：不要做过度设计，哪怕先硬编码（Hardcode）一个返回值来通过测试也可以。
*   **验证**：运行测试，确认它**通过**。

### 3. 🔵 Refactor (重构)
*   **动作**：在测试通过的保护下，优化代码结构。
    *   消除重复代码 (DRY)。
    *   提升可读性。
    *   移除硬编码。
*   **验证**：再次运行测试，确保重构没有破坏功能。

## 异常处理
*   如果现有代码库没有测试框架，优先创建一个最小化的测试环境或脚本。
*   对于 UI 变更，如果难以编写自动化测试，请定义明确的手动验证步骤作为替代。

## 注意事项
*   严禁一次性编写大量代码再回头补测试。
*   时刻关注 `task.md` 的进度，完成一个循环后勾选对应任务。
*   全程使用**简体中文**。
